토스 PO가 기능이 아닌 흐름을 설계하는 이유

1. 동화로 시작하는 메타포

흐름의 본질은 '연결된 생태계'

‘홈’이라는 연못과 다양한 연못(대출, 증권, 카드 등)을 돌아다니는 물고기 이야기

길목이 막히면 이동 불가 → 길을 넓혀주면 다시 흐름이 생김

연못을 연결하는 길과 정화 장치로 생태계(앱) 전체가 건강해짐

Point: 단일 기능보다는 서비스 간 흐름과 연결, 유저 경험 전체를 설계해야 함

2. 문제의 출발점

기능 중심의 수작업 운영, 그 한계

수많은 서비스들이 홈 화면 배너로 노출되기 위해 구글 시트로 요청

운영 리소스 과다, 오류 발생, 스케일 불가

해결: 어드민 툴 → 대시보드 → 노출 로직 → 추천 모델 → CTR 10% 달성

3. 연결을 위한 전략

“코어 내부 광고 지면 대통합 프로젝트”

서비스 추천 배너 구좌를 SDK 형태로 패키징

설치만 하면 효율적인 노출/소재 운영 가능

문제: 제품팀들이 “좋긴 한데 지금은 바빠서...”

전환점: 직접 영업, 맞춤 제안, 설치 기사화, 온보딩 프로그램 제공

4. 조직 개편으로 흐름의 유량 제어

두 팀 분리, 목적별 집중 전략

  • 인텐션스 매니저 팀: 소재 등록 증가, 품질 향상, 팀원이 직접 캠페인 등록·실험
  • 인벤토리 그로스 팀: 클릭률/전환/매출 성과 집중, 지면 위치, 반복주기, 최소 노출 기준 등 실험

4. 조직 개편으로 흐름의 유량 제어 - 성과

성과:

  • 하루 소재 등록 100건 → 500건
  • 소재 노출 900건 → 4,000건 이상

5. 퍼포먼스를 끌어올리는 방법

반복 실험 + ML 모델 도입

유저에게 어떤 서비스를 보여줄지 CTR 예측 모델 도입

지면이 하단으로 밀렸음에도 전환률 & 매출은 약 2배 증가

지표 중심: 전환률, 유효 클릭, 클릭 후 행동 등 다층적 분석

6. 질 관리를 위한 장치들

정화 장치 없이 흐름은 오염된다

다크 패턴, 낚시성 문구, 무관한 소재 → 유저 불신

UT 결과: 불쾌한 소재는 더 이상 이 영역을 탐색하지 않게 만듦

6. 질 관리를 위한 장치들 - 대응

대응:

  • 등록 시 문구 자동 검사
  • 이미지-문구 연결 검토
  • 유저 랜덤 피드백 → 건강 지수화
  • 2초 이내 이탈률, 다음 페이지 진입 등 유효 클릭 분석

7. 메시지 & Q&A

메시지: 기능이 아닌 흐름, 구조가 아닌 연결을 설계하라

어드민은 단순 등록 기능이 아닌, 흐름을 설계하는 출발점

등록 → 운영 → 성과 → 피드백 → 개선이라는 순환 고리를 완성

결국 흐름이 앱의 경험을 만들고, 퍼포먼스를 증명함

Q&A:

Q. 다양한 제품 간 이해관계는 어떻게 조율했는가?

A. 기능 리텐션, 전환 등 각 제품팀 목표에 맞는 성과 정의와 성과 공유, 밀착 커뮤니케이션으로 신뢰 형성

터지는 바이럴 만들기

1. 서론: “바이럴은 왜 어려운가?”

모든 제품이 유저를 원하지만, 확산은 어렵다

토스 앱의 바이럴 제품 사례 소개 (600만, 500만, 400만 명 참여)

핵심 지표는 Viral K (한 명의 유저가 데려오는 유저 수)

목표: K를 높여가며 바이럴을 과학적으로 반복 가능하게 만들기

2. Stage 1: 실패를 각오한 첫 실험

✅ 기존 문법의 한계 → 동그라미 챌린지

일반적인 리퍼럴(초대하면 리워드) → K = 0.2

깨달음: "공유시키는" 것이 아니라 "공유하고 싶게 만들어야 한다"

동그라미 챌린지: 완벽한 원 그리기 → 점수 → 다시 하려면 공유

K = 0.4 (어메이징 수준)

인사이트: 익숙한 구조에서 실패를 두려워하지 않고 첫 번째 실패를 빠르게 만든다

3. Stage 2: 진짜 문제에 접근하다

🔍 본질은 '피초대자가 메시지를 열지 않는 것'

동그라미 챌린지의 공유 성공률 낮음 → 원인 분석

유저의 질문: “읽고 나서 나에게 뭐가 있지?”

해결책: 기대감을 주는 메시지 → 복권 메타 도입

📦 복권 선물하기 제품: 친구에게 복권 선물 → 즉시 확인 가능

천만 원 보상 / 당첨자 수 공개 / 즉시성 강조

K = 0.7, 공유 성공률 +62%

인사이트: “기대감”과 “즉시성”은 메시지 오픈율을 극적으로 끌어올린다

4. Stage 3: 반복을 설계하다

🔁 반복하고 싶은 구조는 따로 있다

문제: 인당 초대자 수 낮음 → 왜 여러 명에게 공유하지 않는가?

분석 결과: 유저가 반복할 이유가 없음 → 반복 메커니즘 필요

한글날 챌린지: 난이도 점진 증가, 아슬아슬한 실패 유도

다시 하려면 공유 → 몰입 + 도전 욕구 자극

K = 0.9, 600만 참여, DAU 천만 돌파

인사이트: “아깝다” “될 것 같다”는 감정은 유저를 반복 행동하게 만든다

5. Stage 4: 모두를 위한 설계

👵 고연령 유저도 반복하게 하려면?

고연령 유저는 공유 취소율↑, 반복↓

원인: 한글 찾기 등 게임 난이도 → 포기 유도

해결책: 노력 기반 기대감 부여 → “가격 할인”이라는 메타 도입

🛍 생필품 에거너시제 (생백시): 생필품 가격을 공유할수록 점점 할인

반복할수록 성취감과 실질 혜택 제공

K = 1.0 달성, 전 연령층 반복 증가

인사이트: 누구나 이해하고 반복 가능한 보편적 메타(할인, 성취)는 성공 확률을 높인다

6. Stage 5: 성공을 반복 가능하게 만드는 법

🧠 실패 요인을 역추적하고 수정하라

문제: “끝까지 가야 보상 받는다” → 유저 탈락

해결 시도: 쿠폰 금액 시각화 (예: 치킨 1만 원)

레벨 시스템 도입: 랜덤 업그레이드 → 목표 예측 불가

유저 간 레벨 공유 문화 생성

결과: 0.9 이상의 K 지속 달성

인사이트: 성공보다 “왜 실패했는가”를 신랄하게 따져야 성공을 반복할 수 있다

7. 핵심 메시지

💡 터지는 바이럴은 구조다

구조의 핵심은 유저 심리, 기대감, 반복 욕구, 즉시 보상

바이럴 성공은 ‘감’이 아닌 정교한 설계와 피드백 루프

“왜 안 될까?”를 끈질기게 묻고, “되려면?”을 상상하라

8. 실행 요약표

단계 실험 핵심 메커니즘 Viral K
1리퍼럴 구조직접적 보상0.2
2동그라미반복 욕구 유발0.4
3복권 선물기대감 + 즉시성0.7
4한글날 챌린지몰입 + 아쉬움 유도0.9
5생백시실익 기반 반복1.0

업계의 통념을 바꾼 하나의 제품

1. 제품 개요: '함께 대출'이란?

토스뱅크 + 제휴 은행이 공동으로 대출 실행

고객은 토스뱅크 앱 하나로 대출 신청·이자 납부·원리금 상환 등 모든 과정을 이용 가능

실제론 두 은행이 각각 반반 대출금을 지급하고 토스뱅크가 대행하여 운영

정의: 공동 대출 구조로 설계된 신용대출 상품

2. 왜 혁신인가?

은행 업계의 통념 3가지 돌파

  • ❌ “은행 간 협업은 불가능하다”
  • ❌ “본질적 업무(심사/실행)는 절대 위탁할 수 없다”
  • ❌ “경쟁 은행 간 제휴는 성립 불가”

→ ✅ ‘함께 대출’은 이를 모두 깼고, 업계 최초로 시도 및 성공

2. 왜 혁신인가? - 성과

성과:

  • 금융위원장상 수상
  • 업계 도입 관심 폭발
  • 금융 당국 정책 참고 사례로 활용

3. 문제 정의와 아이디어의 출발

더 좋은 조건의 대출을 더 많이 제공하려면?

제약: BIS 비율로 인해 자본 대비 대출 가능 한도 제한

당시 자본금 2,500억 → 대출 가능액 2.5조 (시장 규모에 비해 작음)

솔루션: 한 건의 대출을 절반으로 나눠서 타 은행과 공동 실행

자본은 절반만 소요, 공급은 2배 확대 가능

4. 현실의 벽: 규제, 관행, 시스템

시행 전 장애물

  • 규제상 대출 심사·실행은 타 은행 대행 불가
  • 수납, 고객상담도 사례 전무, 구조 설계 불가능처럼 보임
  • 제휴 은행 설득, 당국 공감 확보, 리스크 검토 필요

기회: 고객 중심 사고 → “필요한 가치를 주는 구조라면 시도해볼 만하다”

5. 실행: 제품에서 제도까지 설계

백지에서 그린 새로운 금융 인프라

  • 고객 경험: 하나의 대출처럼 보이게 UI/UX/문서 모두 설계
  • 플로우 설계: 2개의 대출 실행 → 자금 및 메시지 흐름 실시간 동기화
  • 총 개발량: 2,000개 기능, 5개월 테스트, 운영 솔루션 10종
  • 조율 주체: PO가 전 과정(기획–개발–당국–법무–협업) 직접 관리

6. 당국과의 공감대 형성

규제를 혁신으로 전환시키는 방법

  • 20회 이상 대면 미팅
  • 30개 이상 자료 제출
  • 고객 관점 중심의 설득 전략 → 규제 해석 변경 유도

37개월 만에 혁신금융서비스 지정

출시까지 총 45개월 소요

7. 출시 후 확산 전략

단순 출시가 아닌 '업계 영향력 확대'까지

  • 언론 노출 → 사례화, 고객 성공 스토리화
  • 통계 제공 → 누구에게 어떤 금리 혜택이 있었는지 공개
  • 실험: 실행률 개선, 마케팅 확대, 고객군 확장, 진입 장벽 완화

성과: 5개월: 실행 5천억, 9개월: 실행 1조 돌파

8. 왜 성공할 수 있었는가?

구조를 바꾸는 네 가지 원칙

  • 고정관념 없이 상상하기: “원래 안 돼”를 지우고 고객 니즈에서 시작
  • 백지에서 시작해 하나씩 실현하기: 완성형 로드맵 X → 한 걸음씩 실행하며 디벨롭
  • 신뢰 기반 팀워크: 45개월간의 긴 여정, 동료와의 의지와 꾸준함
  • 당국과 공감으로 함께 만들기: “이건 우리가 하고 싶은 것”이 아닌 “이건 고객이 필요로 하는 것”으로 메시지 전달

9. Key Message

업계 통념을 바꾼 제품은 “그럴싸한 아이디어”가 아니라 실제로 가능한 구조로 구현된 실행의 산물이다.

  • 고객의 니즈에 충실할 것
  • 규제와 조직 사이에서 연결고리를 만들 것
  • 도전의 완성은 꾸준함과 공감에서 온다

2. 숨겨진 화면: 금융거래 차단 알림

매일 200~300명은 이 화면을 만난다

FDS (Fraud Detection System): 부정거래 실시간 차단

AML (Anti-Money Laundering): 자금세탁 방지 시스템

이 두 시스템을 토스는 내부에서 직접 설계/구축

이유: 외부 솔루션은 대응 속도, 정밀 탐지력 한계, 1초 송금, 분당 수천 건 거래 대응 불가

3. 보이스피싱: 가장 먼저 막은 것

👶 “엄마, 나 핸드폰 고장 났어”

자녀 사칭 → 신분증 탈취 → 악성 앱 → 송금

탐지 룰 구성: 로그인 24시간 이내, 중국 IP, 100만 원 이상 송금

IP 대역 분석 + 기기 모델명 정규화

→ 피싱 사례 수십 건 사전 차단 성공

4. 끊임없는 진화: 기기 탐지

📱 오래된 중고폰에서 자꾸 사고가 일어난다

피싱범은 2016~2018년 기종을 선호 (아이폰7 등)

토스 내 사용 디바이스 1,054개 전수 분석

모델 코드 → 출시 연도 → 룰 세팅

→ 설정 후 바로 피싱범 검거 성공 사례 발생

5. 로블록스 사고: 차지백 프러드

🎮 “이거 제가 한 결제 아닌데요?”

13세 아이가 14만원짜리 아이템 환불 반복 → 총 1,200만원

정황 파악 → 통화 인터뷰 → 사고 확정

대응: 2회 이상 반복 → 룰 차단, 계정 변경 회피 시도도 탐지하여 대응

러닝: 어뷰징에도 “정의로운 설계”가 필요하다

6. 중고거래 사기: 피해자 → 가해자

💼 “명품가방 75만원, 선착순”

가짜 중고 플랫폼 → 피싱 사이트 → 토스 로그인 유도

사기나라 카페 운영자들과 협업: 실시간 URL/계좌 공유, 슬랙 연동 → 자동 수집

성과: 2개월간 30억 송금 차단

공격자 반격: “토스 막혀요” → 타 앱 유도 문구 등장

7. 송금 UX 내 FDS 통합

🧠 송금 속도 안에 판단을 담는다

계좌번호 입력 → 위험 판단

UX 분기: 레벨1: 완전 차단, 레벨2: 경고 메시지 바텀시트

신규 계좌 여부 탐지: 불변 마트 + 실시간 델타 합산 구조

효율적이고 실시간 반응 가능

8. 실험 기반 UX 고도화

🚨 경고 애니메이션, 딜레이 버튼 도입

사기 의심 계좌: 사이렌 + 딜레이 + 추가 확인

최대 3배 보호 효과

그럼에도 불구하고 송금하는 유저 70~90%

→ 인터랙션 개선이 보호 성과에 직접 기여

9. 송금 이후에도 보호는 계속된다

✅ “토스 잘못이 아니더라도 보상합니다”

안심보상제 (2021~): 송금 후 사고 발생 시 선보상 제공

중고거래 사기 신고창구 운영: 피해자 대응 빠르게 처리

엄마, 나는 커서 ‘다른 은행’이 될래요

1. 문제 발견: “우리는 왜 이 정보를 못 받지?”

규정대로 해도, 데이터는 안 쌓였다

외국인 고객 납세자 정보 수집 화면에서 엉뚱한 자유서술 응답

“나는 학생이라 세금 없다”, “몰라요” 등

시스템은 규정을 따랐지만, 유저는 해석할 수 없었다

비대면에서는 창구 안내자 역할이 없다는 것이 문제

2. 규제, 질문을 바꾸면 열린다

“납세의무가 있나요?” → “있다 치자, 문제 될까?”

오히려 없다고 해도 문제 안 되는 방향으로 UX 재설계

수도승의 비유처럼 같은 질문도 다르게 묻는 것의 힘

핵심: 고객이 ‘잘못된 선택’을 해도 위험이 적은 방향으로 설계

3. 누구를 위한 업무인가?

고객은 국세청일 수도 있다

최종 사용자는 국세청(또는 해외 과세기관)

목적은 조세 회피 방지 → 결국은 '식별자' 수집이 핵심

주민등록번호처럼, 해외 납세자 번호도 자동 제시할 수는 없을까?

4. 진짜 혁신: OECD부터 루마니아 신분증까지

수작업으로 만든 글로벌 납세번호 DB

100개국 이상의 신분증 / 납세번호 체계 수집

직접 유효성 검증 로직도 설계

앱 내 입력값을 국적/체류자격 기반으로 즉시 피드백

→ 디지털 온보딩에 적합한 자동 필터/어드민/알림 연동 완성

5. 국세청을 설득하다

처음엔 긴장 → 나중엔 “다른 은행에도 가이드해야겠네요”

국세청 담당자와의 미팅

처음엔 “왜 이렇게 당연한 걸 묻냐” 반응

→ 토스뱅크가 국가보다 먼저 정리한 사례와 도구를 보여주자

  • 유효성 검증
  • 어드민 연동
  • 실수/이탈 줄이는 UX

반응 반전: “내년에 이 사례 가이드로 쓸 것 같다”

6. 정기검사도 다르게 대한다

“감추는 시간 말고, 배움의 시간으로”

금감원 정기검사에서도 실수 감추기보단 질문으로 접근

“이 질문은 어떤 사고에서 나온 건가요?”

규제기관을 대화 상대로 보고, 공동 설계자로 대함

7. 전략이 된 규제 대응

규제는 리스크가 아니라 운신의 폭을 넓히는 기반

자금세탁방지(AML), 납세 정보 등은 단순 컴플라이언스 항목이 아니라 전략적 자산

RBA (Risk-Based Approach):

→ 단순 체크리스트가 아니라

→ 위험을 정의하고 스스로 해결책을 설계하는 접근

8. 실제 효과와 미래 상상

📈 변화된 성과

  • 보고 대상자 수: 전년 대비 10배 증가 → 현재 3만 명
  • 보고 정확도 및 회수율 대폭 증가: 국적/언어별 안내 메시지 자동화로 열람률↑

팀 내 대화: “이거 다른 은행에도 팔면 안 돼요?”

규제 대응조차 자산화되는 순간

9. Key Message

규제는 ‘해야 하니까’가 아니라 ‘우리가 이걸 왜 하는지’를 묻는 순간, 기회가 된다

  • 고객 중심으로 문제 정의
  • 위험을 판단하고 시나리오화
  • 질문의 방식과 표현을 리디자인

커뮤니티 Growth로 해자 만들기

1. 문제 정의: 왜 커뮤니티인가?

커뮤니티 = 리텐션의 핵심 지표

커뮤니티 사용자의 포스 증권 방문 빈도는 10배 이상

리텐션·도용 빈도·매매 지속률 등 모든 지표에서 커뮤니티 활성 유저가 우수

정체된 MAU (60~70만)를 넘어서기 위해 커뮤니티에 집중

2. 리텐션 유저 vs 이탈 유저 분석

좋은 글이 핵심 아하 모먼트

초성 유저: 실시간 정보 + 심리적 안정감

이탈 유저: 신뢰 불가, “글 대부분 쓸모없다”

인사이트: 좋은 글을 본 경험 → 팔로우 → 리텐션

3. 정량 데이터로 본 핵심 지표

“좋은 글”의 지표는? → 팔로우 유도 글

좋아요, 댓글, 팔로우 중 → 팔로우가 방문 빈도 증분 가장 큼

팔로우 전 조회 글 분석:

  • 뉴스 요약, 차트 분석 등 정보 전달형 글
  • 수익률 인증, 손절 인증 등 진정성 있는 후기

4. Growth 전략: 좋은 글 & 팔로우 증가시키기 (A)

A. 좋은 글 늘리기

  • 도구 제공: 링크 인베드, 투표, 매매 리뷰 모듈 도입 → 매매 리뷰 비율 1% → 20% 이상으로 증가
  • 앰버서더 운영: 외부 인플루언서 초청, 선례 + 소셜 전염 효과 유발

4. Growth 전략: 좋은 글 & 팔로우 증가시키기 (B)

B. 팔로우 늘리기

  • CGC (Company Generated Content): ‘토스증권’ 계정으로 뉴스, 경제지표 요약 → 최초 10만 팔로워 돌파 계정
  • 소셜 프로필 기능: 매매 내역·성과 공개 → 익명 한계 극복 + 신뢰 기반 커뮤니케이션
  • 피드 로직 개선: 팔로우 수 → 글 노출 보정 → 팔로우 전환율 2배 상승

5. 전사 지표로 연결된 성과

커뮤니티 유저가 늘면 증권 전체가 성장한다

  • 커뮤니티 유저 DAU/MAU = 23% → 50% 증가
  • MAU 전체 3배 성장, 커뮤니티 유저 방문 빈도 5~10배
  • 커뮤니티 유저 비율 증가 시 전체 유저 방문 빈도도 동반 상승

6. 해자 효과: 따라할 수 없는 성장 메커니즘

커뮤니티는 단순 기능이 아니라 네트워크 자산

뉴스보다 빠른 정보, 해석, 토론 → 하루 10만 개 글

네트워크 효과로 진입장벽 생성

타사도 따라할 수는 있으나, 동일한 성과는 어렵다

7. 커뮤니티가 유저 생애가치에 끼친 영향

고액 유저도 커뮤니티 파워 유저다

거래액 상위 1% 유저의 행동 10가지 중 4개가 커뮤니티 행동(글, 좋아요, 팔로우 등)

커뮤니티 정지 → 3개월 내 거래량 30% 하락

단순 상관이 아닌 행동 기반 영향력 확인

8. 커뮤니티가 토스증권만의 브랜드가 되기까지

UX 다음으로 커뮤니티가 꼽힌 “차별화 포인트”

UX/UI 다음으로 많은 유저가 ‘커뮤니티’를 언급

이제는 단순 보조 기능이 아니라

“토스증권에 머무는 이유”가 되었다

9. 커뮤니티 Growth의 어려움과 전략

커뮤니티는 닭과 달걀 게임

콘텐츠 소비는 유저 컨트롤 불가

생산 → 소비 → 리텐션 → 확산의 루프 완성 어려움

실험 통해 루프 완성 전략 도출:

10. 소셜 전염 효과: 콘텐츠는 번진다

직접 유도보다, 유저가 유저를 움직이게 하라

예시: 자유게시판 오픈 → ‘암호말’ 게시글 → 관련 유머/짤 생산 폭발

소셜 전염 효과는 콘텐츠 생산을 유도하는 최고의 전략

저희는 실험 안 합니다

1. 왜 실험을 안 하나요?

“실험은 모든 상황에 정답이 아니다”

토스의 문화: 빠른 실험, 빠른 학습

하지만 B2B 광고 플랫폼 ‘토스 애즈’는 특수 상황

  • 광고주는 수가 적고 민감한 사용성 요구
  • 실험 실패 = 고객의 금전적 손해 가능성
  • 벤치마크 기준(기대치)이 높음 → 불완전 MVP 허용 어려움

2. 대신 선택한 전략

“우리는 실험 대신 고객 이해 → 정제된 설계로 간다”

핵심 가치 기반 접근:

  • Customer-centric
  • Focused on impact
  • Fail fast (but internally)

3. 진짜 커스터머 센트릭의 방식

고객군을 나눠 ‘하루의 삶’을 분석

고객 유형: 인하우스 마케터, 광고 대행사 플래너, 내부 운영자 (어드민)

하루 일과 전체 플로우 인터뷰 → 실 사용 맥락 파악

예: 아침 8시에 성과 확인 필요 → 성과 업데이트 시점 리디자인

정산 단위, 계정 구조 → 대시보드 UI 구성 변경

4. “직접 써보자”

팀원들이 직접 사업자 등록 후 광고 집행

핸드폰 케이스 사업 → 광고 집행 → 성과 확인 → 개선

실제 고객 관점에서 데이터 핑퐁, 불편 포인트 파악

주간 스터디로 6개 주요 플랫폼 디테일 비교

5. 실험 없는 MVP 전략

기능은 “최소한으로, 정확하게”

“이 기능 꼭 필요해요?” → 모든 스펙에 질문

예시: 소재 등록 기능

일반적 구현 플로우 (계정 생성 → 조건 설정 → 크리에이티브 업로드 → 승인 시스템)

대신, 초기엔 수동 계정 발급 + 텍스트/이미지만 업로드 → 이메일 승인

5. 실험 없는 MVP 전략 - 결과

결과:

  • 구현 시간 1/4
  • 내부 리소스 2배 효율화
  • 광고 집행 리드타임 70% 단축

6. 성장의 근거: 지표로 말하다

매주 기능 배포 + 사용성 개선 → 폭발적 성장

  • 월간 광고 매출: +220%
  • 사업자 수: 19배

2년간 기능 누적 출시 히스토리:

  • 광고 상품 6종 추가
  • 개인사업자 대응
  • 실시간 전환 보기, 계정 권한 관리 등

7. 구조의 비대화와 리팩토링 전략

하울의 움직이는 성을 리빌드하라

기능 확장 → 코드 복잡도 증가 → 병목 발생

리팩토링 위크 도입 (4~5주 간격)

주요 작업:

  • 계정 트리 구조화 (마더 계정 도입)
  • 캠페인 1만 개 처리 대응
  • 대규모 정산 안정화

8. 사용자 중심 설계의 예시들

제품 기능 스냅샷 (2025년 기준)

  • 8종 광고 상품 관리
  • 실시간 성과 대시보드 + 캠페인 비교 분석
  • 사업자 단위 권한/계정/정산 뷰 제공
  • 정산 상태, 세금계산서, 납부현황 → 한눈에 UX 구성

9. 결론

실험 없이도 제품은 성장할 수 있다

제품의 특성과 사용자에 맞는 성장 전략을 설계한다면,

빠른 실험 없이도 연 매출 2배 성장과 확장 가능한 구조를 만들 수 있다.

토스 팀 전체의 성과를 올리는 제품

1. 문제 정의

팀마다 비슷한 기능을 만들고, 법은 너무 어렵고, 개선은 느리다

서비스 수: 월 5~10개 신규 런칭 / 240여 개 운영 중

1. 문제 정의 - 공통 문제들

  • 동일한 기능(예: 인증, 동의 UI)을 반복 구현
  • 법/가이드 해석이 어려워 불안한 구현
  • 공통 컴포넌트 개선이 각 제품에 반영되지 않음

2. 해결의 키워드: 모듈화 (Modularization)

반복 개발 방지 + 법적 안전성 + 전사적 최적화

하나의 기능을 독립 가능한 ‘블록’으로 설계

2. 해결의 키워드: 모듈화 (Modularization) - 장점

  • 중복 개발 제거
  • 법률/UX 정책 준수 내장
  • 통일된 UI/UX, 운영 효율 증가

메타 포인트: 모듈 하나 개선 = 모든 제품 개선

3. 사례 1: 인증 모듈

SMS, 휴대폰 본인 인증, 1원 인증을 통합

팀마다 별도 개발하던 인증 플로우 → 표준 모듈화

예: 회원가입 / 인증서 발급 → 동일 본인인증 플로우 사용

3. 사례 1: 인증 모듈 - 결과

  • 중복 코드 제거
  • 일관된 사용자 경험
  • 전체 퍼널 효율화 가능

4. 사례 2: 표준 동의 모듈

법률, UX, 라이팅까지 내장한 ‘컴플라이언스 모듈’

제품팀이 법을 몰라도 안전한 동의 설계 가능

타입 다양화: 풀페이지, 바텀시트, 플로팅 버튼 등

사용자에게도 명확한 구조 제공 (예: 이미지, 요약 콘텐츠)

5. 문제: 개선이 어려우면 전체가 낙후된다

모듈도 지속적 개선이 필요하다

“모듈이 전체 서비스 경험을 망치지 않으려면?”

A/B 테스트 & 인터랙션 개선을 내부에서 진행

6. 개선 사례 A: 1원 인증 자동화

기존: 사용자가 직접 은행 선택 → 계좌입력 → 1원 확인

개선: 마이데이터/오픈뱅킹 연결 → 자동 인증

성과: 기존 성공률 60~88% → 자동 인증 성공률 98%

전환율 +35%

향후 API 제한으로 종료, 이후도 다양한 실험 계속 진행

7. 개선 사례 B: 표준 동의 전환율 실험

기존 UI: 선택 + 필수 혼합형

개선 UI: 필수 → 다음 → 선택 분리 + 자연스러운 연동

성과: 선택 동의율 3~5% → 최대 60% 이상 증가

대부분의 서비스에서 전환율 상승 확인

→ 신규 시도 가능성 + 수익 기회 증가

8. 관리 효율도 끌어올리기: 어드민 기반 설계

동의 모듈 어드민화: 배포 없이 서버에서 동의 UI 수정 가능

타이틀, 이미지, 설명 등 커스터마이징 가능

적용 코드도 간소화: 200줄 코드 → 30줄 / 적용 시간 5분 내

9. 임팩트 요약

모듈이 만든 전사적 변화

제품팀은 법을 몰라도, UX를 설계 못해도 안전한 퍼널 생성

모듈 개선 = 전 제품 퍼널 전환율 개선

반복 개발 제거 + 전환율 개선 + 법적 리스크 제거 + 속도 향상

10. Key Message

“우리는 하나의 모듈을 통해, 모든 팀의 성과를 끌어올릴 수 있습니다.”
